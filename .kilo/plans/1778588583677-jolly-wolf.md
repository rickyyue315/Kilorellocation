## 概覽

經過對整個代碼庫的深度分析，發現以下核心問題：

| 問題 | 嚴重性 | 影響範圍 |
|------|--------|----------|
| `business_logic.py` 上帝類（2652行） | **Critical** | 可維護性 |
| `models/` 全部為死代碼（4個TypedDict + 1個Enum） | **High** | 代碼衛生 |

| 模式謂詞在兩處重複實現 | **Medium** | DRY |

## Phase 4: 完善策略模式（P2 — 架構改進）

### 4.1 統一策略構造函數簽名
```python
class BaseMatchStrategy(ABC):
    def __init__(self, *, create_note=None, max_receive_sites=None, **kwargs):
        self._create_note = create_note
        self._max_receive_sites = max_receive_sites
```

### 4.2 提取策略共享的匹配循環模板
將每個策略中重複的「複製 sources → 初始化追蹤 → 迭代過濾 → 轉移」模式提取到基類：
```python
class BaseMatchStrategy(ABC):
    def execute_match(self, sources, destinations, article, product_desc, mode, om):
        temp_sources = [copy.deepcopy(s) for s in sources]
        temp_dests = [copy.deepcopy(d) for d in destinations]
        recommendations = []
        # 由子類實現具體匹配邏輯
        return self._do_match(temp_sources, temp_dests, recommendations, ...)
```

### 4.3 重構 `BSpecialStrategy` — 移除對 `TransferLogic` 的回調依賴
將 `_match_by_priority` 的核心邏輯提取為獨立的 `matching.py` 模塊中的純函數，策略和 `TransferLogic` 都可以調用。

### 4.4 統一 E-mode 處理
將 `E1ModeStrategy` 重命名為 `EModeStrategy`，通過 `cross_om` 參數區分 E1/E1b/E2，消除 `_match_transfers_e_mode` 的內聯邏輯。

### 4.5 統一 `create_note` 使用
`C2ModeStrategy` 和 `SimplifiedSKUStrategy` 應使用基類的 `self._create_note` 而非硬編碼自己的 note 函數。
