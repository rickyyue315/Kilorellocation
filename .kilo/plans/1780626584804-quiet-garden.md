# D2 優化：清貨轉貨(ND限定) — 新增UI選擇鍵 + 限制接收店舖數量 + 放寬數量

## 目標

1. **D2 模式新增 UI 選擇鍵**：讓使用者可切換「不限店舖數量（原設定）」與「限制2間店舖接收（新優化）」
2. **限制模式下**：同一 SKU 多個 ND 轉出店，盡量集中到 ≤2 間 RF 接收店 → 避免轉出店需為大量不同店舖打包
3. **限制模式下**：放寬單一接收店的接收量至原有的 200% → 補償接收店減少的限制
4. **如可能，多款SKU也盡量指向同一接收店**（跨SKU整合 — 較難，列為次要）

### 5. `business_logic.py` — 新增 D2 專用邏輯

#### 5a. `__init__` 接收新參數

```python
def __init__(self, ..., d2_enable_2site_limit: bool = False):
    ...
    self.d2_enable_2site_limit = d2_enable_2site_limit
```

#### 5b. 新增 `_dests_d2_mode(self, group_df)` 方法

與 `_dests_d_mode` 結構相同，但 target_qty / needed_qty 乘上 `D2_NEEDED_QTY_MULTIPLIER`（=2）：

- Priority 1 (緊急缺貨補貨):
  - `target_qty = max(safety_stock * 2, 4)`  — 即 `max(safety_stock, 2) * 2`
  - `needed_qty = max(target_qty - total_available, 0)`
- Priority 2 (潛在缺貨補貨):
  - `target_qty = safety_stock * 2`
  - `needed_qty = max(target_qty - total_available, 0)`

#### 5c. `identify_destinations()` — 條件路由

當 `d2_enable_2site_limit=True` 且 `mode == self.mode_d2` 時，呼叫 `_dests_d2_mode` 而非 `_dests_d_mode`：

```python
def identify_destinations(self, group_df, mode):
    mode_def = self._mode_by_name.get(mode)
    if mode_def and mode_def.dest_method:
        if mode == self.mode_d2 and self.d2_enable_2site_limit:
            return self._dests_d2_mode(group_df)
        method = getattr(self, mode_def.dest_method)
        ...
```

#### 5d. `match_transfers()` — 條件路由

```python
if mode == self.mode_d2:
    if self.d2_enable_2site_limit:
        return _match_d2_mode_impl(self, sources, destinations, article, om, product_desc, mode)
    # 否則 fall through 到一般 matching（無限制的原設定）
```

### 6. `services/matching_engine.py` — 新增 `match_d2_mode()`

新增函式 `match_d2_mode(logic, sources, destinations, article, om, product_desc, mode)`：

- 與 `match_general_mode` 結構相似
- 但每回合 `match_by_priority` 都傳入 `max_receive_sites_per_source=2`
- 只有 **1 輪 Source Priority 1 → Dest Priority 1** 和 **1 輪 Source Priority 1 → Dest Priority 2**  
  （D2 只有 `source_type = 'ND清貨轉出'`、`source_priority = 1`，沒有 RF source rounds）

```python
def match_d2_mode(logic, sources, destinations, article, om, product_desc, mode):
    recommendations = []
    temp_sources, temp_destinations = prep_temp_lists(sources, destinations)
    transfer_sites = set()
    receive_sites = set()
    received_qty_by_site = {}
    source_to_receive_sites = {}
    max_receive = 2

    match_by_priority(logic, temp_sources, temp_destinations, recommendations,
                      article, om, product_desc, 1, 1, transfer_sites, received_qty_by_site, mode,
                      receive_sites=receive_sites,
                      source_to_receive_sites=source_to_receive_sites,
                      max_receive_sites_per_source=max_receive)

    match_by_priority(logic, temp_sources, temp_destinations, recommendations,
                      article, om, product_desc, 1, 2, transfer_sites, received_qty_by_site, mode,
                      receive_sites=receive_sites,
                      source_to_receive_sites=source_to_receive_sites,
                      max_receive_sites_per_source=max_receive)

    return recommendations
```

匯出 `match_d2_mode` 供 `business_logic.py` import。

## 跨SKU整合（目標4）

**「多款SKU都盡量同一目標接收」** 較難實現，原因是目前 matching 是按 `(Article, OM)` 分組逐一執行，組間無互動。

可行的後續方向（本次暫不實作）：
- 在 `generate_transfer_recommendations()` 收集所有 Article 的結果後，再做跨 Article 的 consolidation pass
- 同一 Site 在不同 Article 中都是接收店時，嘗試將相近的 Source 指向同一 Site

此功能需更複雜的全局最佳化演算法，列為未來改進項目。
