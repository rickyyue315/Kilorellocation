# 各模式計算邏輯改善建議

## 總覽

經過完整審閱 25 個轉貨模式的計算邏輯（`business_logic.py`、`services/matching_engine.py`、`strategies/` 下所有策略檔、`models/mode_registry.py`、`config.py`），以下按嚴重程度分為三個等級列出改善建議：

- **P0 — Bug / 邏輯錯誤**：會導致錯誤結果或潛在崩潰
- **P1 — 邏輯缺陷 / 不一致**：不會崩潰但產生次優或不符合預期的結果
- **P2 — 結構性改善**：可維護性、效能、可讀性提升

---

## P0 — Bug / 邏輯錯誤

### 1. 精簡SKU(跨OM) 的 `cross_om_grouping` 未設定，跨OM邏輯為死碼

**位置**：`models/mode_registry.py:124-129`、`strategies/simplified_sku.py:56-60`

**問題**：
`精簡SKU(跨OM)` 的 `ModeDef` 設定了 `cross_om_matching=True` 但**沒有** `cross_om_grouping=True`。這表示分組仍然是 `Article + OM`（同OM分組），所以同一組內的 sources 和 destinations 必然屬於同一個 OM。

然而 `simplified_sku.py:56-60` 中的跨OM約束檢查（Windy→Windy、HD→HA/HB/HC）依賴 `source['om'] != dest['om']`，這在同OM分組下永遠為 `False`，導致這些約束**永遠不會觸發**——是死碼。

**為什麼要改**：
如果用戶選擇「精簡SKU(跨OM)」，他們期望跨OM配對能發生，但實際上系統只在做同OM配對，與「精簡SKU(限同OM)」行為完全相同。

**改善方案**：
```python
# mode_registry.py:124-129
ModeDef("精簡SKU(跨OM)", "精簡SKU(跨OM)", "精簡SKU跨OM轉貨",
        attr_name="mode_simplified_sku_cross",
        families=frozenset({'simplified_sku'}),
        cross_om_grouping=True,   # ← 加入此行
        cross_om_matching=True, source_filter=True,
        strategy_key='simplified_sku',
        source_method='_sources_simplified_sku', dest_method='_dests_simplified_sku'),
```

**風險**：中等。修改後跨OM配對會真正生效，可能產生之前不存在的跨OM調貨建議。需要確認業務邏輯上是否確實需要跨OM，以及 Windy/HD 約束是否足夠。建議先用歷史數據回歸測試。

### 2. E2 模式的 `_make_source` 缺少 `supply_source` 和 `last_2_month_sold_qty` 欄位

**位置**：`strategies/e2_mode.py:13-28`

**問題**：
E2 Phase 3 C-fallback 使用的 `_make_source` 函式是獨立定義的，缺少 `supply_source` 和 `last_2_month_sold_qty` 欄位。而 `business_logic.py:59-76` 中的全域 `_make_source` 包含這些欄位。

如果下游代碼（如 `build_recommendation` 或 `_refresh_recommendation_fields`）嘗試存取這些欄位，會觸發 `KeyError`。

**為什麼要改**：
`strategies/simplified_sku.py:88-90` 中就有檢查 `supply_source` 的邏輯，如果 E2 Phase 3 產生的 source 後續被類似邏輯處理，缺少欄位會導致異常。

**改善方案**：
刪除 `e2_mode.py` 中的獨立 `_make_source`，改為從 `business_logic` 匯入全域版本；或者補齊缺少的欄位：
```python
# e2_mode.py _make_source 補齊
source['supply_source'] = row.get('Supply source', None)
source['last_2_month_sold_qty'] = ...
```

**風險**：低。只是補齊欄位，不影響匹配邏輯本身。

### 3. E2 模式的 `_compute_max_protected_sold` 重複定義

**位置**：`strategies/e2_mode.py:31-37` vs `business_logic.py:104-110`

**問題**：
兩處獨立維護相同邏輯。如果未來修改保護最高銷量店的規則（例如改為 top-2 保護），必須同步修改兩處，否則 E2 Phase 3 的行為會與其他模式不一致。

**改善方案**：
將 `_compute_max_protected_sold` 提取到共用模組（如 `services/utils.py`），兩處統一引用。

**風險**：低。純重構，不改變行為。

## P1 — 邏輯缺陷 / 不一致

### 4. C2 模式匹配不使用多回合優先制，與 C 模式行為不對稱

**位置**：`strategies/c2_mode.py:33-70`

**問題**：
C 模式使用 `match_general_mode`（`matching_engine.py:208-267`），按 ND→RF過剩→RF加強 的順序分 6+ 回合匹配，確保高優先源先配對。

C2 模式的 `C2ModeStrategy` 使用簡單的雙層迴圈（dest→source），源的遍歷順序取決於 `temp_sources` 的原始排列（即 `identify_sources` 的輸出順序）。雖然 sources 在 `_sources_general` 中按 priority 排序，但沒有區分 `source_type`（ND轉出 vs RF過剩轉出 vs RF加強轉出）的回合制匹配。

**為什麼要改**：
C2 作為「C 模式的跨OM版本」，用戶期望匹配優先級與 C 一致。目前 C2 可能出現 RF加強轉出先於 ND轉出配對的情況（如果 ND 排在 RF 之後）。

**改善方案**：
在 `C2ModeStrategy.match()` 中按 source priority 分回合：
```python
for source_priority in [1, 2]:
    priority_sources = [s for s in temp_sources if s['priority'] == source_priority]
    for source_type in [None, 'RF過剩轉出', 'RF加強轉出']:
        # 匹配邏輯...
```
或者考慮直接複用 `match_by_priority` 並加入跨OM支持。

**風險**：中等。改變匹配順序會影響最終配對結果。需要回歸測試確認 C2 的輸出是否符合業務預期。

### 5. C2 模式缺少 `source['site'] in receive_sites` 檢查

**位置**：`strategies/c2_mode.py:37-49`

**問題**：
C2 的匹配迴圈中檢查了 `dest['site'] in transfer_sites`（line 42），但**沒有**檢查 `source['site'] in receive_sites`。這意味著一個在先前配對中已經作為接收方的店鋪，仍可能被選為轉出源。

對比 `can_transfer`（`matching_engine.py:87-88`）：
```python
if source['site'] in receive_sites:
    return False
```

**為什麼要改**：
一個店鋪同時作為接收方和轉出源在物流上不合理，會產生不必要的雙向調貨。

**改善方案**：
```python
# c2_mode.py 在 line 45 後加入
if source['site'] in receive_sites:
    continue
```

**風險**：低。只會減少不合理的配對，不會影響正常配對。

### 6. C1 模式 `C1_MODE_MIN_TRANSFER` 常量未被使用

**位置**：`config.py:45`、`business_logic.py:543`

**問題**：
`config.py` 定義了 `C1_MODE_MIN_TRANSFER = 2`，但在 `business_logic.py:543` 中硬編碼了 `actual_transferable < 2`，沒有引用該常量。

**為什麼要改**：
如果將來需要調整 C1 的最低轉出量，修改 `config.py` 不會生效，造成維護陷阱。

**改善方案**：

```python
# business_logic.py:543
if mode == self.mode_c1 and actual_transferable < C1_MODE_MIN_TRANSFER:
    continue
```

**風險**：極低。純維護性修改。

### 7. D 模式避免1件餘庫存邏輯與後處理單件優化可能衝突

**位置**：`matching_engine.py:32-42`（`_adjust_d_family_remainder`）、`business_logic.py:874-875`（`_optimize_single_piece_transfers`）

**問題**：
D 模式的 `_adjust_d_family_remainder` 在匹配階段調整 transfer_qty 以避免源端留下 1 件餘庫存。而後處理 `_optimize_single_piece_transfers` 則嘗試消除 Transfer Qty=1 的記錄（接收端視角）。

當 D 模式某源只有 1 件可轉出時：
- `_adjust_d_family_remainder` 嘗試 +1 或 -1，但如果 `transferable_qty=1` 且 `transfer_qty=1`，則 `final_remaining=0`（不是1），不會觸發調整
- 後處理會嘗試消除這條 Transfer Qty=1 的記錄

**為什麼要改**：
如果 D 模式的唯一選擇就是轉出 1 件，後處理可能會錯誤地移除這條合理的配對。

**改善方案**：
在 `_optimize_single_piece_transfers` 中加入 D 模式豁免，或在 D 模式下跳過單件優化：
```python
if mode in d_family_modes:
    return recommendations  # D 模式不做單件後處理
```

**風險**：低。D 模式本身已有避免單件的機制，後處理對 D 模式的價值有限。

### 8. B-special 的 `max_receive_sites_per_source` 在 `match_by_priority` 中未直接使用

**位置**：`matching_engine.py:92-96`、`matching_engine.py:137-146`

**問題**：
`match_by_priority` 接受 `max_receive_sites_per_source` 參數（line 146），但函式內部從不直接使用它。限制邏輯在 `can_transfer`（line 92-96）中通過 `logic.b_special_max_receive_sites_per_source`（實例屬性）來檢查。

這意味著：
1. 傳入的 `max_receive_sites_per_source` 參數被忽略
2. `can_transfer` 使用的是全域實例屬性，而非當前調用的具體參數

對比 E1/E2/ND 策略，它們在自己的匹配迴圈中直接使用局部變量 `max_receive_sites_per_source`。

**為什麼要改**：
如果未來需要對不同 B-special 子模式使用不同的 limit 值，現有架構無法支持（因為 `can_transfer` 只讀實例屬性）。

**改善方案**：
在 `can_transfer` 中優先使用傳入的參數：
```python
def can_transfer(..., max_receive_sites_per_source=None):
    limit = max_receive_sites_per_source or logic.b_special_max_receive_sites_per_source
    if limit is not None:
        ...
```

**風險**：低。只是讓參數傳遞鏈更一致，現有行為不變。

### 10. A 模式 single-piece bump 邏輯的 upper_limit 檢查冗餘

**位置**：`business_logic.py:470-476`

**問題**：
```python
if actual_transferable == 1 and remaining_stock >= 3:
    bump_remaining = int(row['SaSa Net Stock']) - 2
    if 2 <= upper_limit:  # ← 此檢查
        actual_transferable = 2
```

`upper_limit = max(int(total_available * 0.2), 2)`（line 458），由於 `A_MODE_MIN_TRANSFER = 2`，`upper_limit` **永遠 >= 2**，所以 `2 <= upper_limit` 永遠為 `True`，是冗餘檢查。

**改善方案**：
移除 `if 2 <= upper_limit:` 檢查，直接執行 bump。

**風險**：極低。不改變行為。

### 11. `match_general_mode` 中 C 模式的匹配回合順序可優化

**位置**：`matching_engine.py:218-267`

**問題**：
C 模式的匹配回合：
```
Round 1: ND→重點補0 (priority 1,1, dest_type='重點補0')
Round 2: ND→Emergency (priority 1,1)           ← 重複匹配 priority (1,1)
Round 3: ND→Potential (priority 1,2)
Round 4: RF過剩→Emergency (priority 2,1, 'RF過剩轉出')
Round 5: RF過剩→Potential (priority 2,2, 'RF過剩轉出')
Round 6: RF過剩→重點補0 (priority 2,1, 'RF過剩轉出', '重點補0')  ← 又回到 priority 1 dest
Round 7: RF加強→Emergency (priority 2,1, 'RF加強轉出')
Round 8: RF加強→Potential (priority 2,2, 'RF加強轉出')
Round 9: RF加強→重點補0 (priority 2,1, 'RF加強轉出', '重點補0')  ← 又回到 priority 1 dest
```

Round 1 和 Round 2 都匹配 `(source_priority=1, dest_priority=1)`，但 Round 1 加了 `dest_type_filter='重點補0'`，Round 2 沒有 filter。由於 Round 1 已消耗了重點補0的 needed_qty，Round 2 中這些 dest 的 `needed_qty` 可能已歸零。但 Round 2 同時匹配 Emergency（也是 priority 1），所以不是完全冗餘。

然而 Round 6 和 Round 9 重新回到 `dest_priority=1`（重點補0），這表示 priority 1 的目的地被分散在三個不同回合中處理。如果重點補0的 needed_qty 在 Round 1 就被滿足，Round 6 和 9 是冗餘的。

**改善方案**：
重新組織 C 模式回合順序，讓每個 dest priority 集中處理：
```python
# ND 優先滿足重點補0
(1, 1, None, '重點補0'),
# ND 滿足 Emergency + Potential
(1, 1, None, None),
(1, 2, None, None),
# RF過剩 優先滿足重點補0
(2, 1, 'RF過剩轉出', '重點補0'),
# RF過剩 滿足 Emergency + Potential
(2, 1, 'RF過剩轉出', None),
(2, 2, 'RF過剩轉出', None),
# RF加強 優先滿足重點補0
(2, 1, 'RF加強轉出', '重點補0'),
# RF加強 滿足 Emergency + Potential
(2, 1, 'RF加強轉出', None),
(2, 2, 'RF加強轉出', None),
```

**風險**：低。只是重新排列現有回合，不引入新邏輯。但需要回歸測試確認配對結果不變。

### 12. E2 Phase 3 C-fallback 的 source 排除邏輯可能遺漏

**位置**：`strategies/e2_mode.py:168-176`

**問題**：
Phase 3 排除 RF source 的條件：
```python
if row['OM'] in e_mode_source_oms:     # 排除 E 模式源所在的 OM
    continue
if row['Site'] in transfer_sites:       # 排除已轉出的站點
    continue
if row['Site'] in receive_sites:        # 排除已接收的站點
    continue
if row['Site'] in non_e_mode_receiving_sites:  # 排除非 E 模式 OM 中的接收站
    continue
```

但 `non_e_mode_receiving_sites`（line 156-157）只包含 `needed_qty > 0` 的目的地。如果某站點在 Phase 1/2 中已被滿足（`needed_qty = 0`），它不會在 `non_e_mode_receiving_sites` 中，但可能在 `receive_sites` 中。

`receive_sites` 檢查（line 173）已覆蓋此情況，所以目前不會出錯。但這種依賴 `receive_sites` 而非 `non_e_mode_receiving_sites` 的邏輯鏈較脆弱，重構時容易打破。

**改善方案**：
添加註解說明 `receive_sites` 和 `non_e_mode_receiving_sites` 的分工，或合併為一個排除集合。

**風險**：極低。目前行為正確，純可維護性改善。

## P2 — 結構性改善

### 13. 各策略的匹配邏輯大量重複

**位置**：`strategies/c2_mode.py`、`strategies/e1_mode.py`、`strategies/e2_mode.py`、`strategies/nd_mode.py`、`strategies/f_mode.py`、`strategies/simplified_sku.py`

**問題**：
每個策略都獨立實現了幾乎相同的約束檢查：
- `source['site'] == dest['site']` → 自身轉讓
- `dest['site'] in transfer_sites` → 源不能是目的地
- `dest.get('rp_type') == 'ND'` → ND 不能接收
- `is_hd_to_hk_restricted(...)` → HD→HK 限制
- Windy→Windy 限制
- `max_receive_qty` 檢查

這些邏輯在 `matching_engine.py:can_transfer` 中已有統一實現，但各策略沒有複用它。

**改善方案**：
提取通用的 `validate_pair(source, dest, ...)` 函式，所有策略統一調用。或者讓所有策略都通過 `match_by_priority` / `can_transfer` 管道，只在需要特殊行為時覆寫。

**風險**：低（純重構），但工作量大，需要仔細測試每個模式。

### 14. `_sources_general` 函式過長（~200 行），建議拆分

**位置**：`business_logic.py:348-554`

**問題**：
`_sources_general` 涵蓋了 A/B/B-special/C/C1/D/D2 所有通用模式的 source 識別邏輯，包含多層 if-elif 分支。函式過長導致難以理解和測試。

**改善方案**：
按模式家族拆分為：
- `_sources_nd_general()` — ND 源識別（所有模式共用）
- `_sources_rf_a_mode()` — A 模式 RF 源
- `_sources_rf_b_mode()` — B 模式 RF 源
- `_sources_rf_c_mode()` — C/C1/C2 模式 RF 源
- `_sources_b_special_type_l()` — B-special Type=L 源

**風險**：低（純重構），但需要確保拆分後的調用順序和狀態傳遞正確。

### 15. 建議加入匹配結果的統計日誌

**位置**：所有策略的 `match()` 方法

**問題**：
目前匹配完成後只有總數日誌（`business_logic.py:993`），缺少每個 Article 組的匹配統計（源數、目的地數、匹配成功數、未滿足需求等）。

```python
logger.debug(f"Article={article}: {len(recommendations)} matches, "

             f"{sum(d['needed_qty'] for d in temp_destinations)} remaining demand")
```

**風險**：極低。只增加日誌輸出。
