# Plan: Mode A 1-Item Transfer Bump Fix

## Problem
In Mode A (`保守轉貨`), under specific conditions, the source's `actual_transferable` computes to **1** because `base_transferable = total_available - safety_stock = 1`. The existing inline bump logic in `_match_by_priority()` (line 2538-2556) intentionally excludes `RF過剩轉出` (the only source type Mode A generates), so 1-piece transfers pass through to the output.

## Root Cause
`business_logic.py:484-491`: `actual_transferable = min(base_transferable, upper_limit, SaSa_Net_Stock)` where `base_transferable` can be 1 when surplus above safety is exactly 1, even though:
- `upper_limit` already has floor 2: `max(int(total_available * 0.2), 2)`
- Safety check `remaining_stock >= safety_stock` passes

## Proposed Fix
Add a post-computation bump in `identify_sources()` for Mode A only, after line 499 (inside the safety check pass block), before the `else: continue` block:

**Condition**: When `actual_transferable == 1` AND `SaSa Net Stock - 1 >= 3` (remaining after 1-piece transfer >= 3), bump to 2 with safety stock relaxed by 1.

**數學等價性證明** (Safety Stock -1 放寬的可行性):
- 原始檢查: `SaSa Net Stock - 1 >= safety_stock`
- 放寬檢查: `SaSa Net Stock - 2 >= safety_stock - 1` ↔ `SaSa Net Stock - 1 >= safety_stock`
- 兩者等價 → 放寬條件不會引入額外風險，只要原始 1 件通過安全檢查，放寬後 2 件必然也通過

**Logic to insert** (after `source_type = 'RF過剩轉出'` at line 499):

```python
# 若轉出僅1件且淨庫存餘3件以上，上調至2件 (避免單件調貨；放寬Safety Stock -1)
if actual_transferable == 1 and remaining_stock >= 3:
    bump_remaining = int(row['SaSa Net Stock']) - 2
    # 放寬安全線: bump_remaining >= safety_stock - 1
    # 等價於 remaining_stock >= safety_stock (原始已通過), 故此處僅檢查上限
    if 2 <= upper_limit:
        actual_transferable = 2
        remaining_stock = bump_remaining
```

`actual_transferable` 會在 line 575 `'transferable_qty': int(actual_transferable)` 自動傳入 source dict，無需額外更新。

## File to Change
- `business_logic.py`: `identify_sources()` method, Mode A block (lines 481-501)

## Impact
- Only Mode A affected
- Safety Stock 放寬 -1 與原始安全檢查數學等價，不降低實際安全水準
- 門檻 `remaining_stock >= 3` → 確保上調至 2 件後來源店仍餘 ≥ 2 件，不會出現只餘 1 件
- Upper limit (20% cap) still respected: check `2 <= upper_limit`

## Edge Cases
- If source's `transferable_qty` is bumped to 2 but destination only needs 1, matching will still send 1 — this is fine because the limitation is on the destination side, not the source
- If `upper_limit = 2` (from the floor), bump to 2 always passes the cap check
- Post-processing `_optimize_single_piece_transfers` (line 2720) remains as fallback for remaining 1-piece edge cases
